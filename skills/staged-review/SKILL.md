# 暂存区代码审查

审查 `git diff --cached` 中的所有变更，输出结构化的审查报告。

## 快速开始

```bash
git diff --cached --stat          # 概览变更文件
git diff --cached                  # 完整 diff
git log --oneline -5              # 最近的提交上下文
```

## 工作流程

### 1. 获取暂存变更

- [ ] 运行 `git diff --cached 2>/dev/null` — 如无暂存变更，提示用户先 `git add`
- [ ] 如果 diff 超过 200 行，先展示 `--stat` 概览，再分文件逐个审查
- [ ] 运行 `git log --oneline -5` 获取上下文

### 2. 逐文件审查

对每个变更文件，按以下维度检查：

- [ ] **逻辑正确性** — 条件分支是否正确？边界条件是否处理？是否有 off-by-one？
- [ ] **安全隐患** — 是否存在注入风险？敏感信息是否硬编码？权限检查是否完备？
- [ ] **性能问题** — 是否有不必要的循环/IO？N+1 查询？大对象分配？
- [ ] **代码风格** — 命名是否一致？函数是否过长？是否遵循项目现有约定？
- [ ] **错误处理** — 错误是否被正确传播？是否有静默吞错？
- [ ] **测试覆盖** — 新逻辑是否有对应测试？边界情况是否测试？

### 3. 输出审查报告

用以下格式输出：

```
## 📋 Code Review: Staged Changes

### 概览
- 文件数: X
- 新增: +Y 行, 删除: -Z 行

### 🔴 严重（必须修复）
- [ ] 问题描述 — 文件:行号 — 修复建议

### 🟡 建议（推荐修改）
- [ ] 问题描述 — 文件:行号 — 改进思路

### 🟢 风格（可选优化）
- [ ] 问题描述 — 文件:行号 — 优化建议

### ✅ 值得注意的好做法
- 值得肯定的代码片段

### 📝 总结
[一句话总结整体质量 + 最需要关注的 1-2 个问题]
```

### 4. 可选：生成报告文件

如果变更较大或用户要求，将报告写入 `.pi/review-{date}.md`。

## 审查重点优先级

1. **逻辑 Bug** > 安全 > 性能 > 风格
2. 新增代码审查深度 > 修改代码 > 删除代码
3. 关注 diff 中 "藏" 在大量上下文中的小改动
4. 对重构类变更：重点检查行为是否保持不变

## 特殊场景

- **二进制文件** — 仅列出文件名，不做审查
- **仅删除代码** — 确认无残留引用
- **大型 diff (>500 行)** — 先出概览，询问用户优先审查哪些文件
- **配置文件** — 检查格式正确性，值是否合理
