# 工作日志

将当前对话中的工作内容综合为结构化的进展记录。

## 流程

1. **综合上下文** — 从对话中提取已完成的工作。不要采访用户，只需综合你已知的内容。

2. **检查当天文件** — 检查 `.pi/worklog/` 下是否已存在当天日期的文件（`YYYY-MM-DD.md`）。如存在，追加到该文件；否则新建。

3. **写入文件** — 使用下方模板。

### 新建

文件路径：`.pi/worklog/YYYY-MM-DD.md`

### 追加

在已有文件末尾添加 `---` 分割线，然后追加新条目。

```markdown
---

## {时间段/简短描述}

{条目内容，使用下方模板}
```

已有文件的开头元信息（日期、项目）无需重复。

## 条目模板

```markdown
## {简短描述}

**时间**: HH:MM - HH:MM（或"本次会话"）
**阶段**: {设计 / 开发 / 测试 / 修复 / 重构 / 文档 / 其它}

### 已完成

- {具体动作 + 结果，尽量可验证}
- 例：`实现 login API，完成 JWT 鉴权`
- 例：`修复 data-processing 模块内存泄漏`

### 进度状态

- 当前进度：{百分比 或 里程碑描述}
- 状态：{on track / delayed / blocked}
- {如有偏差，简述原因}

### 遇到的问题

- **问题**：{具体描述}
  - 影响：{是否阻塞 / 影响范围}
  - 已尝试：{已尝试的解决方案}
- 如无问题，写"无"

### 下一步

- {短期计划}
- 依赖：{需要谁/什么支持，如无则省略}
- 如本次已结束，写"本次工作已完结"

### 决策与变更

- **{决策}**：{理由}
- 只记录有实质影响的决策
- 如无，省略此段

### 关键数据

- {可量化的成果，如性能提升、bug 数、完成量}
- 如无量化数据，省略此段

### 备注

{任何额外观察、风险、技术债、或对未来有价值的参考信息}
```

## 规则

- **不要采访用户** — 直接从对话上下文综合
- **只写已知的** — 不确定的字段宁可省略
- **面向未来读者** — 几周后回顾时应能理解背景和决策理由
- **优先量化** — 有数据就写数据，没有就写描述
- **分割线追加** — 同一天多次调用 `/to-worklog` 时合并到同一文件，用 `---` 分割
