# /to-knowledge

## 快速开始

用户说 `/to-knowledge` 或 "记录一下这些知识" → 扫描对话上下文 → 提取隐知识条目 → 按主题去重合并写入 `.pi/knowledge/` → 输出总结。

## 工作流程

1. **扫描**：回顾当前对话，找出以下特征的隐知识：
   - 做出了技术决策但理由只在对话中说过
   - 提到了坑、边缘情况、容易出错但没写进文档的点
   - 说明了某种约定或模式但代码里没有注释
   - 解释了系统为什么这样工作（而不仅仅是怎样工作）
   - 讨论中暴露出的"以后会忘"的知识点

2. **分类**：为每条知识分配一个分类（见下方分类表）

3. **定题**：确定主题 slug（kebab-case），如 `auth-flow`、`error-handling`、`deployment`。一个主题文件可包含多条相关知识。

4. **去重**：读取 `.pi/knowledge/<slug>.md`，检查是否已有标题/核心点相似的条目。相似则跳过并记录。

5. **合并写入**：新条目追加到已有文件末尾（文件不存在则新建），条目间用 `---` 分隔。

6. **总结**：始终输出写入摘要（见下方格式），不允许静默成功。

## 知识分类

| 分类 | 说明 | 标题前缀示例 |
|------|------|-------------|
| `decision` | 技术/架构决策及其理由 | `decision: 为什么用 JWT 而不是 Session` |
| `gotcha` | 坑、边缘情况、会踩到的雷 | `gotcha: refresh token 过期导致 401 循环` |
| `convention` | 编码模式、命名约定、团队共识 | `convention: 错误码使用三位数字` |
| `insight` | 对系统运作方式的深层理解 | `insight: 中间件执行顺序影响鉴权` |
| `todo` | 已知待填补的文档或代码缺口 | `todo: API 错误响应格式未统一` |

## 条目格式

每个条目使用以下结构：

```markdown
## <category>: <标题>

> **日期**: YYYY-MM-DD | **标签**: tag1, tag2

<正文 — 用简洁的段落解释>
```

示例：

```markdown
## gotcha: Nginx 反向代理时 req.ip 拿到的是 127.0.0.1

> **日期**: 2026-04-28 | **标签**: nginx, express, networking

因为 Nginx 在前面做了反向代理，Express 默认拿到的 req.ip 是 localhost。
需要设置 `app.set('trust proxy', true)` 才能拿到真实 IP。
```

## 主题命名与去重

- **主题文件命名**：`<topic-slug>.md`，放在项目根 `.pi/knowledge/` 下。用简短有意义的 kebab-case，如 `auth-flow.md`、`api-design.md`。避免过于宽泛的名称如 `misc.md`。
- **去重判断**：比较新条目的标题和核心观点是否与已有条目相似。标题相同 → 跳过。观点相同但表述不同 → 可以补充到已有条目中，或标注"已有类似记录"后跳过。
- **合并**：新条目追加在已有内容末尾，保留历史记录。

## 输出总结格式

每次执行后必须输出以下格式的摘要：

```
📝 知识沉淀完成

### 新增

- `.pi/knowledge/<file>.md` — +N 条
  - <category>: <标题>
  - ...

### 跳过（已存在）

- `.pi/knowledge/<file>.md`
  - <category>: <标题>
```

如果无任何新知识可提取，输出：

```
📝 未发现新的隐知识
```

如果所有知识都已存在：

```
📝 所有知识已记录，无需更新
```

## 边界情况

- **上下文不足**：如果对话太短或明显没有隐知识（如只是打招呼），直接说明"对话中未发现可沉淀的隐知识"。
- **主题难以确定**：优先用涉及的核心模块/功能命名。实在无法归类时使用 `general.md`。
- **大规模沉淀**：如果条目超过 10 条，先列出所有条目请用户确认再写入。
